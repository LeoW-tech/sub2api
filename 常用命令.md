# 常用命令

## 当前双机角色

- Linux：主运行面，负责稳定环境、主重启、主运维、主同步提交流程
- Mac：辅助运行面，负责从 GitHub 拉取更新、开发验证、必要时本机运行
- 两边统一入口都是 `./scripts/sub2api-local` 与 `make`
- `stable rebuild` 受脚本约束，必须在 `main` 分支执行；这是脚本行为，不是文档额外规定

## 路径基线

按你当前所在主机选择对应路径：

```bash
# Linux
cd /srv/sub2api/repo

# Mac
cd /Users/meilinwang/Projects/sub2api
```

如果想写成一行命令，后续都按各自主机路径接在后面即可。

## 最常用：重启 stable

如果你现在只是想处理稳定环境，优先看这一段。

```bash
# Linux
cd /srv/sub2api/repo && ./scripts/sub2api-local stable restart
cd /srv/sub2api/repo && ./scripts/sub2api-local stable rebuild

# Mac
cd /Users/meilinwang/Projects/sub2api && ./scripts/sub2api-local stable restart
cd /Users/meilinwang/Projects/sub2api && ./scripts/sub2api-local stable rebuild
```

如果你更习惯 `make`：

```bash
# Linux
cd /srv/sub2api/repo && make stable-restart
cd /srv/sub2api/repo && make stable-rebuild

# Mac
cd /Users/meilinwang/Projects/sub2api && make stable-restart
cd /Users/meilinwang/Projects/sub2api && make stable-rebuild
```

说明：

- `stable restart` 只重启现有稳定环境容器，并等待健康检查通过
- `stable rebuild` 会重建 `sub2api` 镜像，然后自动重新启动 stable；不需要再额外执行一次 `stable up`
- 只要你按当前主机选择正确的 `cd <仓库路径> && ...` 写法，当前终端起始目录不重要
- `stable rebuild` 会检查当前分支必须是 `main`，否则脚本直接拒绝执行
- `stable restart` 不强制要求当前分支是 `main`，但日常仍建议先确认仓库当前就在 `main`

## 统一运维入口

优先使用统一脚本：

```bash

# Mac
cd /Users/meilinwang/Projects/sub2api && ./scripts/sub2api-local <子命令>
```

除非命令本身明确写了 `cd backend && ...` 这类子目录切换，否则不建议在 `runtime/`、`frontend/`、`backend/` 等子目录里直接执行仓库级命令。

## 平台专属守护入口

### Linux: systemd 托管

```bash
cd /srv/sub2api/repo && sudo ./scripts/sub2api-local systemd install
cd /srv/sub2api/repo && ./scripts/sub2api-local systemd status
cd /srv/sub2api/repo && sudo ./scripts/sub2api-local systemd restart
```

说明：

- `systemd install` 用于首次安装或重装 Linux 守护链路
- `systemd status` 查看 `sub2api-stable.service` 与 `sub2api-door-gateway.service`
- `systemd restart` 重新串起 stable 与 `door-gateway`

## 维护类命令

### sync upstream

```bash
# Linux
cd /srv/sub2api/repo && ./scripts/sub2api-local sync upstream

# Mac
cd /Users/meilinwang/Projects/sub2api && ./scripts/sub2api-local sync upstream
```

说明：

- 这条命令两边都能执行，但日常建议优先在 Linux 主运行面执行
- 它会更新 `upstream-main`、推送 `origin/upstream-main`，再从 `main` 切出 `sync/upstream-YYYYMMDD`

### backup runtime

```bash
# Linux
cd /srv/sub2api/repo && ./scripts/sub2api-local backup runtime

# Mac
cd /Users/meilinwang/Projects/sub2api && ./scripts/sub2api-local backup runtime
```

默认情况下，`backup runtime` 会把备份写入当前 runtime 根目录下的 `backups/<timestamp>/`。如需临时改到其他目录，可这样执行：

```bash
# Linux
cd /srv/sub2api/repo && SUB2API_BACKUP_ROOT="$HOME/sub2api-backups" ./scripts/sub2api-local backup runtime

# Mac
cd /Users/meilinwang/Projects/sub2api && SUB2API_BACKUP_ROOT="$HOME/Desktop/sub2api-backups" ./scripts/sub2api-local backup runtime
```

## 日常开发流程

开发用户自己的功能时：

```bash
# Linux
cd /srv/sub2api/repo
git checkout main
git pull --ff-only origin main
git checkout -b feature/<topic>
./scripts/sub2api-local dev rebuild
./scripts/sub2api-local dev up
```

如果希望一整段一次复制执行：

```bash
cd /srv/sub2api/repo && \
git checkout main && \
git pull --ff-only origin main && \
git checkout -b feature/<topic> && \
./scripts/sub2api-local dev rebuild && \
./scripts/sub2api-local dev up
```

完成后：

```bash
cd /srv/sub2api/repo
git add .
git commit -m "feat: <topic>"
git push -u origin feature/<topic>
```

如果希望整段复制执行：

```bash
cd /srv/sub2api/repo && \
git add . && \
git commit -m "feat: <topic>" && \
git push -u origin feature/<topic>
```

如果用户没有明确要求推送，只提交本地 git 即可。

## 同步原始仓库

同步上游时，统一使用：

```bash
cd /srv/sub2api/repo && ./scripts/sub2api-local sync upstream
```

该流程会：

1. `fetch upstream` 和 `fetch origin`
2. 更新 `upstream-main`
3. 推送 `origin/upstream-main`
4. 从 `main` 切出新的 `sync/upstream-YYYYMMDD`
5. 把 `upstream-main` 合入这个同步分支

同步后建议继续：

```bash
cd /srv/sub2api/repo && ./scripts/sub2api-local dev rebuild
cd /srv/sub2api/repo && ./scripts/sub2api-local dev up
```

验证通过后，再把同步分支合回 `main`。如果这次同步需要让 Mac 跟进，再回到上面的“双机同步场景”执行 `fetch/pull`。
